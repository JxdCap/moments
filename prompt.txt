基于当前项目实际状态执行本任务。不要重写项目，不要新开第二套实现，不要推翻现有 React + Vite + TypeScript + CSS Modules + PocketBase 结构；在现有代码基础上做“第二轮功能完善 + 温和收骨架”。

目标：保留当前这版已经做出来的朋友圈首版页面与整体方向，把它修到更接近官方微信朋友圈的交互体验，并把文件数量从当前过多的状态温和收缩到更适合维护的规模。重点是：公开点赞、图片灯箱、移动端响应式、评论交互改造、温和合并文件。不要扩展与本轮无关的新功能。

一、本轮必须完成的功能完善
1. 移动端响应式优化
- 改为 mobile-first 思路
- 去掉当前过强的“桌面中间摆一个大白卡/手机壳”的感觉
- 在常见手机宽度下，页面应自然铺满、留白合理、卡片尺寸与字体更接近朋友圈
- 优化封面、头像、正文、九宫格、评论区、操作按钮在窄屏下的间距和尺寸
- 桌面端保持可浏览，但优先保证手机宽度下观感

2. 图片灯箱预览
- 点击动态图片后进入全屏黑底预览
- 支持查看该条动态内的多张图片
- 支持关闭
- 显示当前索引，例如 2 / 6
- 打开预览时禁止背景滚动
- 不要引入第三方 UI 库
- 实现尽量贴近朋友圈式沉浸浏览，而不是简单弹窗

3. 公开点赞
- 允许公开点赞
- 每条动态支持点赞状态与点赞数展示
- UI 和交互节奏尽量接近朋友圈
- 若当前没有完整后端规则，可先在 demo / 本地状态里跑通，并为 PocketBase 预留清晰对接位
- 不要做复杂用户体系；公开点赞可以采用轻量方案，但页面交互必须成立
- 点赞入口放到操作菜单中，而不是直接把大按钮铺在页面上

4. 评论交互改造
- 移除“每条动态默认常驻评论输入框”的交互
- 默认状态下，每条动态只展示已有评论列表，不直接展示输入框
- 每条动态底部改成更接近官方朋友圈的操作入口：右下角三个点或等价操作按钮
- 点击操作按钮后弹出一个小操作层
- 操作层中至少有：点赞、评论
- 点“评论”后，再展开该条动态的评论输入框
- 提交评论成功后，允许自动收起，或保持展开但必须自然
- 不要把评论交互做成独立后台表单页

二、温和收骨架
1. 当前文件数量过多，请做“温和合并”，但不要压缩成难维护的大泥球。
2. 目标是把整体文件规模收敛到更适合继续迭代的程度；原则上尽量收敛到约 15～20 个核心源文件左右，允许略有浮动，只要明显比现在精简即可。
3. 合并原则：
- 可以把过小、强耦合、只服务于单个卡片的小组件温和并入更高层组件
- 可以把很薄的映射层、类型层做适度合并
- 可以把文章卡片、音乐卡片、视频块这类极小组件合并进动态卡片组件，前提是不会把单文件做得失控
- 不要把所有逻辑都塞进 App.tsx
- 不要把所有样式都塞进一个超长 global.css
- 不要为了减少文件数而牺牲清晰度
4. 建议保留清晰的分层：App / Timeline / MomentCard / CommentForm / Lightbox / posts service / comments service / global styles 等核心结构。
5. 删除或合并后，要同步清理无用文件、无用导出、无用样式和无用类型。

三、保持现有边界，不要扩展
1. 继续保持单页面应用
2. 继续保持单用户站点
3. 不要做多用户注册系统
4. 不要改成 Next.js、Nuxt、SSR、全栈 CMS
5. 不要引入 Tailwind、shadcn、任何 UI 组件库
6. 不要加入私信、好友系统、无关后台大面板
7. 不要增加网页端发布窗口；发布仍然留给 iOS 快捷指令或外部接口
8. 不要在本轮加入与任务无关的新内容类型
9. 不要把 PocketBase superuser 凭证放进公开前端

四、实现要求
1. 优先复用现有项目已有的代码和样式，做增量改造
2. 样式仍使用原生 CSS / CSS Modules
3. 交互逻辑尽量清晰，不要在一个文件里堆过多杂乱状态
4. 图片灯箱、点赞/评论菜单、评论展开状态、移动端适配要真正跑通
5. 对于 PocketBase 未完全接通的部分，可以做清晰可替换的本地实现，但要保留明确的 service 对接位
6. 点赞和评论都要考虑 demo 模式与 PocketBase 模式的兼容
7. 三个点操作菜单的视觉和位置尽量接近朋友圈，不要做成普通后台按钮组

五、执行与验证
1. 修改完成后必须执行一次可验证检查，至少运行 npm run build
2. 若有必要，可补充最少量的脚本或类型修正，但不要扩大工程建设范围
3. 输出结果时明确汇报：
- 本轮修改了哪些文件
- 本轮删除/合并了哪些文件
- 当前文件数量相较修改前减少了多少
- 已完成了哪些功能完善
- 哪些部分仍是 demo 实现、后续应对接哪个 service / 接口
- 构建结果

六、优先级
本轮优先级从高到低：
1. 三个点操作菜单 + 评论交互改造
2. 图片灯箱
3. 移动端响应式
4. 公开点赞
5. 温和收骨架

按以上要求直接在当前仓库落地。重点是“在现有可用首版上修成交互更像朋友圈、结构更克制的第二版”，不是做新项目，也不是无限扩功能。
