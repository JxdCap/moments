基于当前项目实际状态执行本任务。不要重写项目，不要新开第二套实现，不要推翻现有 React + Vite + TypeScript + CSS Modules + PocketBase 结构；在现有代码基础上做“第四轮：用户体验增强与好玩功能补强”。

目标：在当前已经具备朋友圈外观、公开点赞、灯箱、评论交互、移动端适配的基础上，继续提升“普通访客第一次打开就愿意继续玩”的感觉。本轮重点不是继续扩展工程骨架，而是补足几个真正有使用感、好玩、适合个人朋友圈站的增强功能。

一、本轮必须完成的功能

1. 评论身份记忆
- 访客第一次评论后，将昵称、邮箱、网址保存到本地
- 后续再次打开页面或再次评论时，自动回填这些字段
- 邮箱仍然只用于 Gravatar，不在公开界面展示
- 要有清晰的本地存储实现，不要接复杂用户体系
- 不要破坏现有评论校验逻辑

2. 评论成功反馈优化
- 当前评论提交成功后的反馈要更自然、更像产品，而不是后台表单提示
- 评论提交成功后，给出轻量成功提示
- 若当前为 demo 模式，可直接把新评论自然插入当前评论区，保持页面反馈即时
- 若当前为 PocketBase 模式且存在审核状态，可用更自然的提示文案说明“已提交，待显示”之类，不要做成僵硬系统提示
- 评论框提交成功后可自动收起，但整体体验要自然

3. tags 轻量筛选
- 利用现有 tags 数据，在页面上增加轻量筛选能力
- 不要做复杂搜索系统，不要做独立搜索页
- 推荐做法：在时间流顶部或合适位置增加一个轻量 tags 筛选条/筛选区
- 支持：查看全部、按单个 tag 过滤、清除筛选
- 样式要克制，不能做成博客标签云或后台筛选器
- 被选中的 tag 状态要清晰
- 过滤逻辑要与当前时间流自然融合

4. 相册视图 / 照片模式
- 在不改成多页面的前提下，增加一个轻量“照片模式”或“相册视图”
- 目标是让访客可以更方便地浏览所有图片动态，而不是只能一条条刷时间流
- 可以做成：顶部切换入口、局部模式切换、抽屉式面板、覆盖层等，但不要改成完整图库子系统
- 相册模式中，优先展示图片动态中的图片，布局适合快速浏览
- 点击图片后仍然复用现有灯箱能力
- 保持整体风格仍属于朋友圈站，而不是独立图片网站

5. 灯箱补一个“原图 / 下载”轻量能力
- 在现有灯箱基础上，增加非常轻量的“查看原图”或“下载图片”能力，二选一或都支持均可
- 不要做复杂菜单，不要破坏现有沉浸感
- 按钮视觉要弱，不要喧宾夺主
- 保留现有关闭、索引、切换逻辑
- 不要引入第三方 UI 库

二、尽量完成的增强项（若实现成本低，可一并做）
1. 时间流顶部增加“全部 / 照片”这种很轻的模式切换，前提是不破坏现有页面节奏
2. 让评论成功后的插入、提示、收起动作更连贯
3. 让 tag 筛选状态与相册模式切换之间的关系清晰，不互相打架

三、明确不要做的事
1. 不要重构成多页面
2. 不要引入 Tailwind、shadcn、任何 UI 组件库
3. 不要改成 Next.js、Nuxt、SSR、全栈 CMS
4. 不要新增复杂后台管理系统
5. 不要继续大规模收骨架或压文件数
6. 不要重做作者登录入口；保留当前作者登录入口现状
7. 不要引入公开用户注册/登录体系
8. 不要做复杂搜索引擎、复杂推荐系统、复杂动画系统
9. 不要让“照片模式”变成一个完全独立的新产品

四、实现要求
1. 优先复用现有 posts / comments / lightbox / timeline 相关结构
2. 样式仍使用原生 CSS / CSS Modules
3. 本轮重点是用户体验增强，不是工程展示
4. 本地身份记忆建议使用 localStorage
5. tags 筛选与相册模式要真正可用，不要只做静态占位
6. demo 模式与 PocketBase 模式都要尽量兼容
7. 若需要少量新增状态，请保持清晰，不要把大量状态杂糅到单一超长文件中
8. 若需要新增少量组件或 hooks，可以新增，但要克制，不要再次把仓库铺大

五、执行与验证
1. 修改完成后至少运行一次 npm run build
2. 输出时明确汇报：
- 本轮修改了哪些文件
- 评论身份记忆如何实现
- 评论成功反馈做了哪些优化
- tags 筛选如何使用
- 相册视图/照片模式如何进入和退出
- 灯箱新增了什么轻量能力
- 构建结果

六、优先级
本轮优先级从高到低：
1. 评论身份记忆
2. 评论成功反馈优化
3. tags 轻量筛选
4. 相册视图 / 照片模式
5. 灯箱原图 / 下载能力

按以上要求直接在当前仓库落地。重点是：保留当前已经成型的朋友圈站，在不破坏现有结构的前提下，补一批真正能让访客觉得“更顺手、更好玩”的增强功能。
